# Notion导入与素材上传网络优化（2026-03-05）

## 1. 背景与问题现象

### 1.1 现象
- 管理后台 `import-notion` 请求偶发超时或失败，前端看到 `502 Bad Gateway`。
- 素材图片批量上传在网络波动时耗时明显偏长，偶发整批失败。
- 本地调试时可能开启系统代理，线上部署在阿里云公网机，网络路径与本地不同，行为不稳定。

### 1.2 影响
- Notion 导入成功率不稳定，导入体验差。
- 图片上传等待时间长，影响文章生产效率。
- 故障排查成本高，错误信息不够可诊断。

### 1.3 页面/抓包证据说明
- 管理后台上传文章页面，`POST /api/admin/articles/import-notion` 发生 `502`。
- Network 中请求耗时长，网关返回超时/上游不可达类错误。

## 2. 根因分析

### 2.1 Notion 出网链路
- 后端默认网络模式偏向直连，未充分适配代理或系统网络环境差异。
- Notion 相关调用（导入、OAuth、公开页面）未完全复用同一套超时/重试/代理策略。

### 2.2 Notion 重复抓取
- “先预览再导入”流程中，对同一链接常出现重复抓取，增加等待和失败概率。

### 2.3 图片上传同步外部调用
- 上传流程中包含同步反向地理解析（reverse geocode），外部网络慢时会阻塞上传主链路。

### 2.4 前端上传批次策略可优化
- 部分页面一次提交大量文件或大体积表单，容易触发网关限制或偶发失败。
- 缺少按批次重试和更细粒度失败提示。

## 3. 设计方案

### 3.1 Notion 统一网络客户端
- 抽离统一 Notion HTTP 客户端，覆盖：
  - Notion API 导入请求；
  - Notion OAuth token 交换；
  - Notion 公开页面导入请求。
- 统一超时、重试、错误分类与代理解析。
- 网络模式默认 `AUTO`，解析顺序固定：
  1. `HTTPS_PROXY` / `HTTP_PROXY` 环境变量；
  2. JVM / System Proxy（`ProxySelector`）；
  3. direct 直连。

### 3.2 预览缓存复用
- 引入服务端短期缓存（TTL + 最大条目数）。
- 缓存键：`user + shareUrl + authMode + tokenHash + notionVersion`。
- `preview` 写缓存；`import` 优先读缓存，未命中再抓取。

### 3.3 上传链路非阻塞化
- 素材上传主链路调整为：写盘 + EXIF 时间解析优先返回。
- reverse geocode 改后台异步执行，短超时，失败不影响上传成功。
- API 返回结构保持兼容，`province/city` 允许为空。

### 3.4 前端上传与导入优化
- 统一 API 默认超时，并按场景覆写（Notion 导入、上传使用更长超时）。
- Notion 预览/导入增加有限重试与错误分级提示。
- 素材上传统一为“分批 + 并发 + 每批总字节限制 + 批次重试”。

## 4. 配置清单

### 4.1 后端环境变量（示例）
- `NOTION_NETWORK_MODE=AUTO`
- `NOTION_CONNECT_TIMEOUT_MS=5000`
- `NOTION_READ_TIMEOUT_MS=45000`
- `NOTION_MAX_RETRIES=3`
- `NOTION_PREVIEW_CACHE_TTL_SECONDS=600`
- `NOTION_PREVIEW_CACHE_MAX_ENTRIES=200`
- 可选代理：
  - `HTTPS_PROXY=http://<proxy-host>:<proxy-port>`
  - `HTTP_PROXY=http://<proxy-host>:<proxy-port>`

### 4.2 Nginx 建议（示例）
```nginx
client_max_body_size 50m;

location /api/ {
    proxy_pass http://127.0.0.1:8080/api/;
    proxy_connect_timeout 15s;
    proxy_send_timeout 300s;
    proxy_read_timeout 300s;
}
```

### 4.3 本地开发变量（示例）
- `apps/spa/.env.local`
```bash
VITE_API_BASE_URL=http://localhost:8080/api
```

## 6. 回滚方案

### 6.1 代码回滚
- 回滚到上一个稳定发布版本（`/opt/blog/releases/<sha>`）。
- 执行 `rollback_release.sh <sha>` 并验证健康检查。

### 6.2 配置回滚
- 将新增网络与缓存配置恢复为旧值。
- 需要时切回 `NOTION_NETWORK_MODE=DIRECT`（不推荐长期）。
- 恢复 Nginx 代理超时配置后重载：
  - `nginx -t && systemctl reload nginx`

### 6.3 验证
- 回滚后验证：
  - `GET /api/categories` 健康检查通过；
  - 后台导入与上传核心路径可用；
  - 日志中无持续异常堆积。
