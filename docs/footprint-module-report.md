# 足迹页面功能开发报告

## 1. 文档目的

本文用于归档已下线的“足迹页面”功能范围、前后端结构、交互价值与可复用设计资产，供后续重做旅行日志、城市回访、生活编年或地图叙事类模块时复用。

当前结论：

- 足迹模块已于 `2026-03-07` 从前端路由、后台管理、后端接口与部署流程中移除。
- 与文章封面复用的素材能力被保留，并收口为独立的 `cover-materials` 能力。
- 数据库中的 `travel_journey`、`travel_plan`、`travel_plan_task`、`footprint_location` 已被判定为可删除的旧表；`footprint_photo` 因仍承担封面素材存储职责而保留。

## 2. 原模块定位

足迹页面不是简单的“旅行照片展示页”，它原本承担了 4 个目标：

1. 对外展示已经发生的旅行章节，把零散地点和照片组织成公开可读的内容。
2. 对内管理未来计划，把灵感、预算、同行人、任务清单集中沉淀。
3. 把照片上传、地址绑定、旅程归档串成一条运营链路。
4. 为博客提供一类非文章型内容入口，让博客不只围绕技术文章展开。

## 3. 已实现的功能点

### 3.1 前台足迹页

- 顶部 Hero 区
  - 旅行编年馆标题、副标题、模块说明
  - 旅程数量、照片数量、到访城市、覆盖年份 4 个概览指标
- 时间线筛选
  - 按年份筛选
  - 按关键词筛选
  - 按标签筛选
  - 仅查看含照片旅程
- 旅程时间线
  - 按年份分组
  - 每条旅程展示标题、日期区间、城市、摘要、照片数
  - 点击卡片切换右侧详情
- 旅程详情侧栏
  - 封面图
  - 标题、起止日期
  - 正文或摘要
  - 照片数、同行人、预算区间
  - 标签
  - 位置概览
  - 照片宫格
- 登录态差异
  - 未登录用户只能看公开旅程
  - 已登录用户可额外看到未来计划预览区

### 3.2 后台足迹管理

- 旅程管理
  - 新建、编辑、删除旅程
  - 标题、时间、摘要、正文、标签、同行人、预算、封面维护
- 计划管理
  - 新建、编辑、删除未来计划
  - 状态、优先级、预算、标签、备注、任务清单维护
  - 支持“完成并归档”为旅程
- 照片资产管理
  - 批量上传
  - 待处理照片分页
  - 地址绑定
  - 地址桶归档到旅程
  - 从旅程中取出地址桶
  - 单桶拖拽归档与多选批量归档

### 3.3 后端接口与数据结构

- 公开接口
  - `GET /api/journeys`
  - `GET /api/journeys/years`
  - `GET /api/journeys/{id}`
- 登录态接口
  - `GET /api/travel-plans`
- 后台维护接口
  - 旅程增删改、位置绑定/解绑
  - 计划增删改、状态更新、完成归档
  - 素材上传、待处理列表、地址绑定、归档处理
- 核心表
  - `travel_journey`
  - `travel_plan`
  - `travel_plan_task`
  - `footprint_location`
  - `footprint_photo`
## 4. 原设计的优点

- 内容层次完整：公开回忆、私有计划、后台资产链路是一条闭环。
- 情绪感较强：Hero、时间线、章节详情的叙事感比普通列表页强。
- 有长期扩展空间：天然可以扩展成地图、城市回访、路线专题、旅行文章联动。
- 运营视角合理：先上传图片，再绑定地址，再归档旅程，符合内容沉淀路径。

## 5. 暴露出的逻辑与产品问题

### 5.1 结构过重

- 对当前博客主线来说，足迹模块占用了过多导航与后台心智。
- 它带来了独立的数据模型、接口、权限、部署脚本和迁移脚本，维护成本明显高于实际使用频率。

### 5.2 展示语义不够稳定

- 概览卡展示的是当前筛选结果，但文案容易让人误以为是全量统计。
- 旅程时间线与右侧详情是强绑定结构，移动端和长期增长后不够弹性。

### 5.3 数据口径不够稳

- 前端曾有静默去重逻辑，可能吞掉真实存在的重复旅程。
- 年份归档依赖单一日期口径，跨年旅程会被错误归档到单一年份。

### 5.4 长期运营成本偏高

- 旅程、地点、照片、计划之间存在强耦合。
- 部署脚本一旦继续自动执行旧迁移，会把废弃结构重新带回生产环境。

## 6. 本次下线后的保留资产

以下能力仍然保留，且已从足迹语义中抽离：

- 文章封面素材池
- 封面图上传与删除
- 文章 `coverPhotoId` / `coverImage`
- 后台素材池菜单 `COVER_MATERIALS`
- `footprint_photo` 表作为封面素材底层存储

这意味着：未来如果要重做旅行模块，不需要重新发明图片素材基础设施，只需要在现有素材池之上构建新的内容模型。

## 7. 如果未来重做，建议的产品方向

### 方向 A：轻量“生活归档”

- 不做独立计划系统
- 只保留时间线 + 城市标签 + 图片 + 短文
- 更适合博客型站点

### 方向 B：旅行章节

- 保留旅程概念
- 去掉复杂的地址桶绑定和计划任务系统
- 每段旅程直接作为“章节卡片 + 图片组 + 文章联动”

### 方向 C：城市回访档案

- 不以旅程为主轴，而以城市为主轴
- 每个城市下面挂多次到访记录、照片、相关文章
- 更容易长期增长，也更适合“重复去一个地方”的内容沉淀

## 8. 推荐的重做原则

1. 不要一次重建“旅程 + 地点 + 计划 + 资产流水线”全套模型。
2. 优先保留对外可见、最有表达价值的部分。
3. 所有统计都必须明确是“全量”还是“当前筛选结果”。
4. 默认假设数据会持续增长，必须内建分页、分组、截断或折叠策略。
5. 后台维护模型要比前台展示模型更简单，而不是更重。
6. 任何需要额外迁移脚本和独立权限体系的子系统，都要先验证复用价值。
## 9. 可复用提示词

### 9.1 产品规划提示词

```text
你是一个资深产品经理和内容系统设计师。请为一个长期运营的个人博客设计“旅行/生活编年模块”。

要求：
1. 以博客为主，不要做成独立旅游网站。
2. 优先强调内容沉淀、可回看、可持续增长，而不是一次性炫技。
3. 默认数据会越来越多，因此首页、归档页、时间线都必须有分组、分页、截断或切换策略。
4. 后台维护成本要低，避免过重的数据模型和过多状态流转。
5. 输出内容包括：模块定位、信息架构、前台页面、后台页面、数据模型、长期演进建议。

请给出 3 套复杂度不同的方案：轻量版、标准版、扩展版，并说明取舍。
```

## 10. 与本次下线对应的交付

- 删除前台足迹路由与页面
- 删除后台足迹管理与素材墙页面
- 删除 `journey / travel-plan / footprint` 相关控制器、服务、DTO、实体、仓储
- 保留封面素材池并独立成 `cover-materials`
- 更新部署脚本，不再自动执行旧足迹迁移 SQL
- 新增删表脚本：`backend/scripts/drop-footprint-module-tables.sql`

## 11. 结论

足迹模块并不是失败的功能，它的问题主要在于“对于当前博客主线来说太重”。这次下线的核心收益不是少一个页面，而是把系统边界重新收紧到更适合长期维护的范围内。

如果未来重做，建议从“轻量旅行章节”或“城市回访档案”起步，而不是恢复原来的全量模型。
